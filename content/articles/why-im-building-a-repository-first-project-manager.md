## The problem

Most project management tools are built around organizations first and code second. That can work for teams, but it often feels heavy for independent developers.

My real source of truth is usually the repository: code, issues, notes, docs, branches, releases, experiments, and all the small decisions that accumulate while building.

## The idea

Drift starts from the repository and builds outward.

Instead of asking a developer to mirror their work into another system, Drift should make the existing project easier to understand and move forward.

## What repository-first means

- Tasks belong close to the work they describe
- Docs should sit beside implementation context
- Project status should emerge from real files and decisions
- Planning should support shipping, not replace it

## The bet

The bet is that project management can feel lighter when it respects how developers already think.

Drift is still early, but the direction is clear: make the repository feel like a complete workspace instead of only a code container.
